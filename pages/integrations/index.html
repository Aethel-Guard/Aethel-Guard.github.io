// Integration Details

Every connection, documented

Protocol, direction, data flow, and setup guide for each integration.

📱

Vigilar ↔ Vigilar Guard

Consumer threat detection pipeline
REST API WebSocket OpenAPI 3.1 API Key Auth

Vigilar clients (iOS, Android, Desktop) send security events to Guard's FastAPI server. Guard runs Sigma detection rules and Ember AI triage, then returns triaged incidents with severity, confidence, and recommended actions. High-severity incidents trigger a WebSocket push for real-time alerting.

POST/api/v1/eventsSubmit security events → Guard
POST/api/v1/incidentsGuard → client (triaged incident)
GET/api/v1/recommendationsFetch remediation steps
WS/ws/alertsReal-time alert push (high severity)

Direction: Bidirectional — events flow up, incidents and alerts flow back
Auth: X-API-Key header
Setup: See documentation →

🔗

Vigilar Guard ↔ Wazuh

SIEM/XDR bidirectional webhook integration
Webhook Port 8085 Bidirectional Network ACL auth

Guard listens on port 8085 for incoming Wazuh webhooks — no API key required, authenticated by network ACL. Wazuh sends alert webhooks to Guard for AI triage. Guard forwards its own detections to Wazuh for XDR correlation and additional rule matching.

HOOKPOST :8085/webhooks/wazuhWazuh → Guard (incoming alerts)
POSTWazuh APIGuard → Wazuh (forward detections)

Direction: Bidirectional — Wazuh alerts → Guard, Guard detections → Wazuh
Auth: Network ACL (no API key on webhook endpoint)
Setup: See documentation →

🧠

Vigilar Guard → OpenCTI

Threat intelligence sharing via STIX 2.1
STIX 2.1 REST API One-way push

When Guard's Ember AI triage identifies IoCs, TTPs, or threat actor references, it structures the intelligence as a STIX 2.1 bundle — observables, indicators, relationships, and attack patterns — and pushes it to OpenCTI via its REST API. This enriched threat intel is then available to all SOC analysts.

STIXPOST /v2/stix-bundlesGuard → OpenCTI (threat intel)

Direction: Guard → OpenCTI (one-way push)
Data: STIX 2.1 bundles — observables, indicators, relationships, attack patterns
Setup: See documentation →

🔄

Vigilar Guard → MISP

Collaborative threat intel sharing via PyMISP
PyMISP One-way push Community sharing

Guard shares detection events with MISP communities using the PyMISP Python library. Events are created, tagged, and pushed to configured MISP instances for collaborative threat intelligence sharing across organizations and communities.

POSTPyMISP APIGuard → MISP (event creation + tagging)

Direction: Guard → MISP (one-way push)
Data: Detection events with IoCs, tags, and context
Setup: See documentation →

📡

Vigilar Guard → Splunk / QRadar / ArcSight

SIEM forwarding via CEF over syslog
CEF Syslog One-way push Industry standard

Guard formats alerts as Common Event Format (CEF) messages and forwards them over syslog to any compatible SIEM — Splunk, IBM QRadar, or Micro Focus ArcSight. This uses industry-standard protocols so no custom integration is needed on the SIEM side.

CEFsyslog://siem-host:514Guard → SIEM (CEF alerts)

Direction: Guard → SIEM (one-way push)
Data: CEF-formatted alerts over syslog (port 514)
Setup: See documentation →

🤖

AI SOC Team ↔ Vigilar Guard

Direct detection event delivery to SOC agents
Webhook Direct delivery Bypass Wazuh

For low-latency triage, Guard can deliver detection events directly to AI SOC Team agents via webhook, bypassing the Wazuh SIEM layer. The Alpha (detection) agent picks up alerts, queries Wazuh for full context, and dispatches to Bravo (response), Charlie (intel), or Delta (forensics) as needed.

HOOKPOST /soc/webhooks/guardGuard → SOC (direct detection delivery)
POSTGuard APISOC → Guard (query incident context)

Direction: Guard → SOC (direct or via Wazuh)
Data: Detection events with full triage context
Setup: See documentation →

Ember Core → All Products

Unified AI inference via FastAPI on port 8081
FastAPI Port 8081 OpenAI-compatible Selene P1-P7 guards

Ember Core serves as the unified inference layer for all Aethel Guard products. It exposes an OpenAI-compatible API on port 8081 via FastAPI, with Selene P1-P7 safety guards enforced at inference time. GET:8081/healthHealth check

Direction: All products → Ember Core (inference requests)

// Data Flows

How data moves

Three primary data flow paths through the Aethel Guard ecosystem.

Flow 1 — Consumer Threat Detection (Vigilar → Guard → Vigilar)

1
Vigilar detects anomalous activity on-device

Security event captured locally — no PII leaves the device. Event structured as JSON with type, severity, and context.

2
POST /api/v1/events → Vigilar Guard

Event sent with X-API-Key authentication. Guard validates against OpenAPI 3.1 contract.

3
Guard runs Sigma detection rules + Ember AI triage

Sigma engine matches against 8 rule categories. Ember Core (port 8081) provides AI triage with product-specific system prompt. Selene P1-P7 guards enforce safety at inference.

4
Guard returns incident via POST /api/v1/incidents

Triaged incident with severity, confidence, and recommended actions. WebSocket pushes real-time alert if severity is high.

5
Vigilar displays remediation guidance to user

Recommendations fetched via GET /api/v1/recommendations. User sees actionable steps. No raw threat data stored on-device beyond the session.

Flow 2 — Enterprise Detection Pipeline (Guard → SIEM → AI SOC Team)

1
Vigilar Guard detects threat via Sigma rules or AI triage

Detection triggered by incoming event stream or Wazuh webhook on port 8085. Guard enriches with threat intel from OpenCTI.

2
Guard forwards detection via CEF/syslog to Wazuh

Alert formatted as CEF message, sent over syslog to Wazuh SIEM. Alternatively, direct webhook to AI SOC Team for bypass.

3
Wazuh indexes alert and runs additional detection rules

Wazuh XDR correlates with existing alerts, applies its own rule set, and enriches with endpoint telemetry.

4
AI SOC Team Alpha agent picks up alert from Wazuh

Alpha (detection agent) queries Wazuh API, pulls full alert context, and begins triage. Ember Core provides inference on port 8081.

5
Alpha dispatches to Bravo (response) or Charlie (intel)

If actionable, Bravo creates TheHive case and triggers Shuffle SOAR playbook. Charlie enriches with OpenCTI/MISP threat intel.

6
Delta (forensics) conducts post-incident analysis

Forensic artifacts collected, OpenSearch indexed, Loki logs queried. Grafana dashboard updated with incident metrics.

Flow 3 — Threat Intel Sharing (Guard → OpenCTI → MISP → SOC)

1
Guard enriches detection with threat context

Ember AI triage identifies IoCs, TTPs, and threat actor references during analysis.

2
Guard pushes STIX 2.1 bundle to OpenCTI

Structured threat intelligence — observables, indicators, relationships, and attack patterns — pushed via OpenCTI REST API.

3
Guard shares events via PyMISP to MISP

Detection events shared with MISP communities for collaborative threat intel. PyMISP API handles event creation and tagging.

4
AI SOC Team Charlie agent queries OpenCTI + MISP

Charlie (threat intel agent) pulls enriched context from OpenCTI and MISP, correlates with active incidents, and feeds back to Alpha and Bravo.

5
Enriched intel displayed on Grafana dashboards

Threat intel metrics, sharing stats, and correlation results visualized in Grafana. OpenSearch provides full-text search across all intel.

Vigilar Guard API

Key endpoints for the detection engine. Full OpenAPI 3.1 spec available in the repository.

Authentication

All endpoints require an API key passed in the X-API-Key header.

HTTP
# Request headers
X-API-Key: vg_live_your_api_key_here
Content-Type: application/json

Submit Events

Send security events from Vigilar or external systems to Guard for detection and triage.

POST /api/v1/events
POST /api/v1/events
Content-Type: application/json
X-API-Key: vg_live_your_api_key_here

{
  "event_type": "anomalous_process",
  "severity": "medium",
  "source": "vigilar_ios",
  "context": {
    "process": "unknown_bin",
    "pid": 1337
  }
}

// Response: 202 Accepted
{
  "event_id": "evt_01HZX...",
  "status": "queued_for_triage",
  "estimated_triage_ms": 2500
}

Receive Incidents

Guard delivers triaged incidents back to the client. High-severity incidents also trigger a WebSocket push.

POST /api/v1/incidents (Guard → Client)

Content-Type: application/json

  "severity": "high",
  "confidence": 0.87,

  "recommendations": [

    "Block network access for unknown_bin"

// Open Standards

No vendor lock-in. Ever.

Every integration uses established open standards. If you switch SIEMs, change threat intel platforms, or build your own pipeline — Aethel Guard speaks your language.

📐

STIX 2.1

OASIS Standard · JSON

Structured Threat Information eXpression. Guard pushes observables, indicators, relationships, and attack patterns to OpenCTI in STIX 2.1 format. Industry-standard threat intel serialization.

📋

CEF

Common Event Format · ArcSight

Common Event Format. Guard formats alerts as CEF messages for forwarding to Splunk, QRadar, ArcSight, or any CEF-compatible SIEM over syslog. No custom parser needed.

📨

Syslog

RFC 5424 · UDP/TCP 514

Standard logging protocol. Guard forwards CEF alerts over syslog to any SIEM. No proprietary transport — just the standard syslog protocol every SIEM already ingests.

🔌

REST API

OpenAPI 3.1 · JSON

Every Guard endpoint is documented in OpenAPI 3.1. REST + JSON over HTTPS with API key authentication. Standard HTTP methods — POST to submit, GET to query, no custom protocol.

⚡

WebSocket

RFC 6455 · WSS

Real-time bidirectional communication for alert delivery. Guard pushes high-severity incidents over a persistent WebSocket connection — no polling, no webhook configuration needed.

🔐

OAuth2

RFC 6749 · Bearer tokens

OAuth2 bearer token support for external API access. Guard authenticates to OpenCTI, MISP, and other external services using OAuth2 — no hardcoded API keys where standards-based auth is available.
